GIT: flujos de trabajo

Created by @patokeilty

Comparando flujos de trabajo

Vamos a presentar los modelos de trabajo mas usuales con repositorios basados en Git.

De ninguna manera estos son los únicos posibles, son expuestos aquí a manera de guía.

  • Centralizado (alla SVN)
  • Feature branches
  • Gitflow

Centralizado

  • provee repo central como copia autorizada (truth source)
  • su flujo de trabajo recuerda a SVN
  • todos los devs tienen una copia local
  • ventaja: permite aislar al desarrollador, commitea local hasta que debe compartir sus cambios (a través del repo centralizado o upstream)
  • único branch de nombre master, similar al trunk de svn

Centralizado (cont.)

Clona el repositorio, queda ubicado en la rama principal 


$ git clone git:git@servername:port/repo.git
$ git checkout master

En sus copias locales edita código, y commitea 


$ git commit -m "New code from Bob, first commit."
$ git commit -m "Fixed some issue, second commit."

Finalmente pushea ambos cambios al repositorio central, y estos quedan listos para compartir con el equipo 


$ git push origin master

Centralizado (cont.)

Resolviendo conflictos

Va a ocurrrir que en determinado momento los commits de un developer van a diverger del repositorio central, y no va a poder pushearlos!

Para poder publicar sus commits, necesita traer los cambios del repo central y rebasear sus cambios sobre lo último. El resultado es una historia perfectamente lineal (como en SVN).


$ git pull --rebase origin master

Feature branches

  • emplea repo central como copia autorizada
  • en vez de commitear siempre a master, se crea rama para la nueva feature en que se va a trabajar
  • los nombres de las ramas deben ser descriptivas: issue-#21, new-header
  • ventaja: permite mayor independencia al desarrollador, permite compartir el feature en que está trabajando sin necesidad de publicar a la rama master.
  • para llevar los cambios de la rama de la feature a la principal o master, se emplea un mecanismo conocido como pull request

Feature branches (cont.)

Pull request

Básicamente expresa el deseo del desarrollador de juntar (mergear) los cambios de su feature a la rama principal, entonces se aprovecha como mecanismo de revisión del código, previamente a ser integrada con master.

Una vez que el pull request ha sido aprobado (sea cual fuere la manera), la publicación de esa feature es similar al caso del workflow centralizado.

Feature branches (cont.)

Ejemplo

  • Bob inicia un nuevo feature branch 
    
$ git checkout -b new-feature master
  • Bob trabaja en la feature ... 
    
$ git status
...
$ git add 
$ git commit
  • cuando cree ha terminado 
    
$ git push -u origin new-feature
  • Bob emite un pull request utilizando la herramienta provista (GitLabs, Stash, Github)

Feature branches (cont.)

Ejemplo

  • Alice es notificada del nuevo pull request, ella examina y comenta el código de Bob. Luego de varias idas y vueltas con nutrido intercambio de ideas, parecen estar de acuerdo en integrar lo nuevo a master.
  • alguien debe mergear a master y pushear los cambios 
    
$ git checkout master
$ git pull
$ git pull origin new-feature (git merge new-feature)
...
$ git push

Gitflow

  • es un flujo de trabajo con git propuesto por Vincent Driessen at nvie
  • muy difundido y empleado en la actualidad
  • un poco mas complejo que los anteriores ya que establece varias prácticas y conceptos comunes: stable branch(master), branch integración(develop), release branches (release/NN), hotfix(hotfix/id), feature branches (feature/id)

Gitflow (cont.)
